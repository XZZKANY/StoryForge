# Cursor-like Project Workbench UI Plan

生成时间：2026-06-15 +08:00

## 目标

StoryForge 主工作台采用类似 Cursor 的项目式 IDE 布局：

- 左侧叫“项目”，负责项目列表与项目切换。
- 中间是 AI 交互区，负责讨论、生成建议、审查、修订协调和上下文解释。
- 右侧是当前项目的文件工作区，负责展示大纲、人物、设定、正文等文件树，并打开具体文件查看或编辑。

核心心智：用户不是在和一个聊天框聊天，而是在一个小说项目里，让 AI 协助理解、修改和推进项目文件。

## 设计原则

1. **项目优先，不叫项目树。**
   左上角入口直接叫“项目”，与 Cursor 的侧栏项目区对齐。用户先选择一本书或一个故事项目，再进入项目内部文件。

2. **AI 居中，但不取代项目。**
   中间区域是 AI 协作主场，但 AI 的每一次回复都必须围绕右侧打开的项目文件、选中文本或左侧选中的项目上下文。

3. **右侧承载作品事实。**
   大纲、人物、设定、正文、审查报告、伏笔表等都作为项目文件或文件夹呈现。AI 结果必须能落到这里，而不是只停留在聊天记录里。

4. **像 IDE，不像内容后台。**
   UI 密度、分栏、文件标签、折叠目录、打开状态、上下文引用都应接近 Cursor/VS Code，而不是仪表盘卡片页。

5. **动作隐藏在工作流里。**
   “生成 / 审查 / 修订 / 继续”不是一级按钮矩阵，而是从 AI 输入、选中文件、选中文本、命令面板或建议补丁中自然触发。

## 信息架构

```text
┌──────────────┬──────────────────────────────┬────────────────────────────────────┐
│ 项目          │ AI 交互                      │ 文件工作区                            │
│              │                              │                                    │
│ 雾港回声      │ 当前项目会话                  │ 《雾港回声》文件树                    │
│ 雪线以北      │ 当前打开文件上下文              │  大纲/总纲.md                         │
│ 第七封来信    │ 讨论 / 审查 / 建议补丁          │  大纲/第02章节点.md                   │
│              │                              │  人物/周眠.md                         │
│              │                              │  设定/旧电台.md                       │
│              │                              │  正文/第02章.md                       │
└──────────────┴──────────────────────────────┴────────────────────────────────────┘
```

## 左侧：项目

左侧只负责项目级选择，不展开具体的大纲、人物、设定文件。

应包含：

- 顶部标题：`项目`，带下拉箭头。
- 工具图标：折叠、更多、新建/打开项目。
- 项目列表：项目名、状态、最近更新时间或阶段。
- 当前选中项目高亮。

示例：

```text
项目 ˅        ↗  …  文件夹+

▣ 雾港回声
  长篇 / 修订中

▢ 雪线以北
  新书 / 大纲期

▢ 第七封来信
  短篇 / 待审
```

非目标：

- 不在左侧展开章节树。
- 不把大纲、人物、设定直接挂在全局项目栏下面。
- 不让左侧承担文件编辑或审查详情。

## 中间：AI 交互区

中间是用户与 AI 协作的主区域。

应包含：

- 当前项目会话标题，例如 `《雾港回声》项目会话`。
- 当前上下文条，例如 `大纲/第02章节点.md`、`人物/周眠.md`、`设定/旧电台.md`。
- 对话流。
- AI 建议补丁或待确认操作。
- 底部输入框。

AI 回复应显式引用右侧文件：

- “建议修改 `大纲/第02章节点.md`。”
- “这会影响 `正文/第02章.md`，是否同步？”
- “需要查看 `设定/旧电台.md` 后再判断。”

中间区常用动作：

- 审查当前文件。
- 修改当前文件。
- 打开关联文件。
- 生成建议补丁。
- 将建议应用到右侧文件。

## 右侧：文件工作区

右侧是当前项目内部结构和内容落点。

应包含两层：

1. 项目文件树。
2. 文件查看/编辑区。

文件树示例：

```text
《雾港回声》

大纲
  总纲.md
  第02章节点.md
  反转表.md

人物
  周眠.md
  林砚.md
  沈确.md

设定
  雾港.md
  旧电台.md
  潮汐塔.md

正文
  第01章.md
  第02章.md
  第03章.md

质量
  审查报告.md
  伏笔表.md
  版本记录.md
```

文件查看区应支持：

- 打开文件标签页。
- 当前文件路径展示。
- 文件内容查看或编辑。
- AI 旁注。
- 关联文件。
- 版本记录入口。
- 应用/撤销 AI 建议的视觉状态。

## 焦点模式

默认是三栏并排，但保留类似 Cursor 的焦点切换。

### 默认模式

```text
项目 | AI 交互 | 文件树 + 文件内容
```

适合日常创作、审查、浏览项目。

### AI 聚焦

```text
项目 | AI 交互最大化 | 文件预览
```

适合讨论构思、拆大纲、审查复杂问题。

### 文件聚焦

```text
项目 | AI 线索栏 | 文件树 + 文件内容最大化
```

适合沉浸写作、阅读大纲、编辑正文。

## 交互流程

### 打开项目

1. 用户在左侧 `项目` 列表选择 `雾港回声`。
2. 右侧加载该项目文件树。
3. 中间 AI 会话切到该项目上下文。
4. 默认打开最近文件或项目首页文件。

### 审查当前文件

1. 用户在右侧打开 `正文/第02章.md`。
2. 中间 AI 上下文条显示当前文件。
3. 用户输入“审查这一章节奏”。
4. AI 输出问题、证据位置和建议补丁。
5. 用户确认后，补丁写入右侧文件或生成待应用 diff。

### 从大纲同步正文

1. 用户打开 `大纲/第02章节点.md`。
2. AI 建议调整节点。
3. 用户确认写入大纲文件。
4. AI 提醒是否同步影响 `正文/第02章.md`。
5. 用户选择同步、跳过或打开正文对照。

## 实施阶段

### Stage 1：文档与原型收口

目标：固定信息架构，不继续发散布局。

任务：

- 确认三栏定义：左项目、中 AI、右文件。
- 将当前原型从“正文编辑区”调整为“文件工作区”。
- 删除或标注过时命名：`项目树`、`正文/大纲编辑区`。
- 保留 `?prototype=storyforge-ui` 作为临时验证入口。

验收：

- 页面首屏能清楚表达“项目 IDE”。
- 左侧只显示项目列表。
- 右侧显示项目内文件树。
- 中间 AI 明确引用右侧文件上下文。

### Stage 2：真实 Desktop IDE 对齐

目标：把 Web 原型决策迁移到 Desktop IDE 主体验。

任务：

- Desktop 左栏改为项目入口与项目列表。
- Desktop 右栏承载真实本地文件树。
- 中间 Assistant 读取当前项目、当前文件、选中文本作为上下文。
- 与 Tauri 文件系统能力对接：打开项目、读取文件、保存文件。

验收：

- 能打开一个本地 StoryForge 项目目录。
- 能在右侧浏览 `大纲/人物/设定/正文` 文件。
- 能选中文件后在中间问 AI。
- AI 回复能引用当前文件路径。

### Stage 3：AI 建议落地到文件

目标：让 AI 不只是聊天，而是能对项目文件产生可确认修改。

任务：

- AI 输出结构化建议补丁。
- 文件工作区展示待应用 diff。
- 用户可接受、拒绝或只保存为旁注。
- 写回文件后保留版本记录。

验收：

- 对 `大纲/第02章节点.md` 的建议能生成 diff。
- 用户确认后文件内容发生变化。
- 可撤回或查看历史版本。

### Stage 4：项目级能力增强

目标：让项目结构支持长篇创作管理。

任务：

- 增加项目模板：长篇、短篇、系列、拆文库。
- 增加文件类型语义：大纲、人物、设定、正文、质量、导出。
- 增加关联文件索引：人物出场章节、设定引用位置、伏笔绑定章节。
- 增加命令面板：打开文件、审查当前文件、同步大纲到正文、导出。

验收：

- 文件树不只是目录展示，而能表达创作语义。
- AI 可以基于文件类型选择不同审查策略。
- 用户能从项目级视角管理一本书。

## 非目标

- 不把 StoryForge 做成纯聊天首页。
- 不把“生成 / 审查 / 修订 / 继续”做成主界面大按钮。
- 不让左侧项目栏承担文件树职责。
- 不在 Web 上继续做大规模主体验投入；Web 原型只用于验证 UI 决策。
- 不在第一阶段实现复杂版本控制、多人协作或完整发布流程。

## 当前结论

StoryForge 的目标 UI 是：

```text
左边：项目
中间：AI 交互
右边：项目文件树 + 文件查看/编辑
```

一句话：

StoryForge 应该像 Cursor 一样让用户先进入项目，再让 AI 围绕项目文件工作；AI 是协作层，文件才是作品事实源。
