# Claude-like Agent Conversation Panel

生成时间：2026-06-18 +08:00

## 背景

StoryForge 桌面端采用项目式 IDE 布局：

```text
左侧：项目
中间：Agent 对话
右侧：项目文件树 + 文件查看/编辑
```

本文定义中间 Agent 对话区的产品形态。目标不是做一个按钮驱动的任务面板，而是做一个类似 Claude 的创作会话：用户通过自然语言与 StoryForge 讨论、审稿、构思、确认方向；只有涉及写入文件、生成 diff、导出等动作时，才需要显式确认。

## 目标

中间 Agent 对话区应成为 StoryForge 的创作协作主场：

- 用户通过对话表达创作意图，而不是先选择工具按钮。
- Agent 先交流、追问、说明口径，再行动。
- 审稿、讨论、方案、草稿都在对话中自然推进。
- 文件写入、diff 生成、导出必须经过用户确认。
- 会话具有短期记忆，项目事实必须确认后再沉淀。

一句话：

StoryForge 中间区应该像一个懂网文的编辑坐在旁边聊，而不是后台任务调度器在跑流水线。

## 非目标

- 不把中间区做成重型 `Agent Run` 步骤面板。
- 不用“开始审查”“执行任务”等按钮作为主要入口。
- 不在顶部展示模型、模式、项目路径等执行配置。
- 不让 Agent 在未经确认时写入项目文件。
- 不把普通分析、审稿、讨论都包装成可视化流程卡片。

## 信息架构

```text
┌──────────────────────────────────────────────┐
│ ConversationHeader                            │
│ 第三章节奏审查                           ...  │
├──────────────────────────────────────────────┤
│ MessageList                                   │
│                                              │
│ StoryForge                                   │
│ 我看完了。第三章的问题不是慢，而是期待被打断。 │
│                                              │
│                         审一下第三章节奏       │
│                                              │
│ StoryForge                                   │
│ 可以。我先按商业爽点节奏看。                 │
│                                              │
├──────────────────────────────────────────────┤
│ ComposerBox                                   │
│ 输入想法、问题，或 @ 引用上下文...             │
│ +  @ 当前文件   StoryForge · Claude · 编辑模式 ↑ │
└──────────────────────────────────────────────┘
```

组件边界：

```text
ConversationPanel
├─ ConversationHeader
├─ MessageList
├─ LightweightStatus
└─ ComposerBox
```

## ConversationHeader

顶部只表达会话身份。

应包含：

- 自动生成的对话标题。
- 新会话默认标题：`新的创作会话`。
- 用户第一轮或前几轮后生成标题，例如：
  - `第三章节奏审查`
  - `反派动机重构`
  - `第十二章续写方向`
  - `女主人物弧光修正`
  - `开篇黄金三章优化`
- 右侧只放轻量入口，例如 `...`，用于后续重命名、历史、删除、新会话等。

不应包含：

- 模型名称。
- 创作模式。
- 项目路径。
- 当前文件路径。
- 大块运行状态。

原因：

用户进入中间区时的心理模型是“我正在一个创作会话里”，顶部应定义会话，而不是定义系统状态。模型与模式影响的是下一条发送，因此应该靠近输入框。

## MessageList

消息区采用 Claude 式阅读体验，而不是即时通信式密集气泡。

### Agent 消息

Agent 消息是左侧正文块。

规则：

- 显示名称：`StoryForge`。
- 不使用头像。
- 不使用强边框大卡片。
- 适合长文本、分段、列表、审稿报告。
- 最大阅读宽度建议约 720-800px。
- 回复应有判断，而不是只堆清单。

示例：

```text
StoryForge
我看完了。第三章的问题不是慢，而是“期待转移太突然”。

主要问题：

1. 第 9 段开始解释设定，压掉了上一章延续的危机。
2. 主角的主动选择出现得晚，读者会感觉她被剧情推着走。
3. 章尾钩子只有信息，没有新的压力。

我建议先改第二点，因为它会顺手带动节奏。
你想让我先给局部改法，还是直接重排这一节？
```

### 用户消息

用户消息是右侧小气泡。

规则：

- 不显示“你”。
- 不显示用户名称。
- 不使用头像。
- 最大宽度建议约 60%-70%。
- 背景比页面略亮即可，不要用强蓝色。
- 文本内容即为用户刚说的话。

示例：

```text
                                      审一下第三章，看看节奏是不是拖了
```

### 状态消息

读取、记忆、执行状态应轻量展示，不做重面板。

可作为 Agent 消息中的一行，也可作为浅灰色系统行。

示例：

```text
正在读取：第三章.md、大纲.md、人物设定.md
```

```text
本轮沿用：商业爽点、不文学腔
```

不建议：

```text
Agent Run
- 理解目标并制定步骤
- 扫描项目上下文
- 读取当前稿件
- 调用 Orchestrator
- 等待确认
```

这些步骤可以存在于内部状态或日志里，但不应成为中间区主视觉。

## ComposerBox

底部输入框是主入口。

结构：

```text
┌──────────────────────────────────────────────┐
│ 输入想法、问题，或 @ 引用上下文...             │
│                                              │
│ +    @ 当前文件    StoryForge · Claude · 编辑模式    ↑ │
└──────────────────────────────────────────────┘
```

规则：

- 输入框本体最大，鼓励用户直接说话。
- `+` 用于添加上下文、附件或文件。
- `@ 当前文件` 表示下一轮会引用右侧打开文件。
- 模型和模式放在右下，影响下一条消息。
- 发送按钮使用小圆形上箭头。
- `Ctrl+Enter` 可作为快捷发送，但不应成为显眼文案。
- `/技能` 可以支持，但不作为主按钮展示；用户输入 `/` 时再弹出技能菜单。

模型、模式不放顶部的原因：

- 模型与模式属于“下一次发送配置”。
- 会话标题属于“当前对话身份”。
- 二者不应混在同一层级。

## Agent 行为规则

核心判断：

Agent 先判断用户是在“聊想法”，还是“要结果”。

### 1. 聊想法

用户输入：

```text
这个女主是不是太软了？
```

Agent 不应立刻改稿。应先一起判断：

```text
有一点，但不是性格软，是她在关键场景里缺少主动选择。
你想让她变得更强势，还是保持温柔但更有主心骨？
```

### 2. 要结果

用户输入：

```text
审一下第三章节奏
```

Agent 可以直接进入分析，但先说明口径：

```text
可以。我按商业连载节奏看，重点检查冲突进入、信息密度和章尾钩子。
我会参考当前章和本卷大纲。
```

### 3. 意图模糊时，只追问一个关键问题

不连续审问用户。

示例：

```text
你更想看人物动机，还是章节节奏？
```

### 4. 意图清楚时，先说明口径

示例：

```text
我按“读者是否愿意继续翻页”的标准来看。
```

### 5. 需要上下文时，自然说明要看什么

示例：

```text
我会先看当前章、前文摘要和人物设定。
```

### 6. 结果要给判断

不只罗列问题，应给出判断。

示例：

```text
问题不是慢，而是期待被打断。
```

### 7. 每次结果后用语言推动下一步

不依赖按钮继续。

示例：

```text
如果你想保住这段设定，我建议走 A；如果你想让章尾更抓人，我建议走 B。
你更想保哪边？
```

## 写入确认规则

所有创作决策都通过对话发生，只有写入文件才需要显式确认。

不需要确认的行为：

- 分析问题。
- 审稿意见。
- 构思方向。
- 给修改方案。
- 给一段不写回文件的草稿。

需要确认的行为：

- 写入原文件。
- 生成待应用 diff。
- 导出交付稿。
- 修改人物卡、设定、世界观、剧情事实。
- 将会话判断沉淀到项目记忆。

示例：

```text
StoryForge
可以。我建议先动三处：压缩设定解释、提前主角选择、把章尾钩子改成直接威胁。
如果你认可，我先给一版 diff，不直接写回原文。
```

用户可以用自然语言确认：

```text
行，给我一版 diff
```

或：

```text
先别改，继续拆问题
```

## 记忆规则

记忆分两层。

### 会话记忆

会话记忆在当前对话里自动延续。

适合自动记住：

- 用户这一轮偏好的审稿口径。
- 当前任务目标。
- 临时风格偏好。
- 用户刚刚否定或认可的方向。

示例：

```text
我会在这次会话里继续按“商业爽点、不文学腔”的标准看。
```

### 项目记忆

项目记忆必须确认后写入。

必须确认的内容：

- 人物设定。
- 世界观规则。
- 剧情事实。
- 长期风格约束。
- 作品级禁忌或偏好。

示例：

```text
这更像人物设定。要不要记到女主人物卡里？
以后写她时，我会按“温柔但有主心骨，关键场景必须有主动选择”处理。
```

原则：

- 会话偏好可以自动延续。
- 作品事实不能擅自沉淀。
- 写入项目资料前必须说明写到哪里。

## 隐藏状态模型

UI 不直接展示复杂状态机，但 Agent 内部可按以下状态组织行为：

```text
discussion → analysis → proposal → drafting → confirmation
```

含义：

- `discussion`：聊天、构思、澄清意图。
- `analysis`：审稿、拆问题、读取上下文。
- `proposal`：提出修改方案或创作方向。
- `drafting`：生成草稿、修订文本或 diff。
- `confirmation`：等待用户确认写入、保存、导出或沉淀记忆。

这些状态应体现在 Agent 的自然语言中，而不是暴露为任务看板。

## MVP 实施边界

第一版只需要完成体验形态，不必一次实现完整智能体能力。

MVP 应包含：

- `ConversationHeader`：显示自动生成或默认会话标题。
- `MessageList`：Agent 左侧正文块，用户右侧小气泡。
- `ComposerBox`：Claude 式底部输入栏。
- 轻状态行：读取上下文、沿用偏好等。
- 发送按钮文案从 `Agent 执行` 降级为图标发送。
- 原有后端能力可以保留，但 UI 上不再把步骤面板作为主视觉。

MVP 可暂缓：

- 真正的 LLM 标题生成。
- 完整会话历史管理。
- 长期项目记忆写入。
- `/技能` 弹出菜单。
- 模型切换弹层。
- 多文件上下文选择器。

## 与现有桌面端的关系

当前桌面端中间区已有 `ChatWindow`，但偏向 Agent Run 执行面板：

- 有运行步骤卡片。
- 输入按钮偏“执行任务”。
- 消息更像工具回显。

后续实现时，应把 `ChatWindow` 改造成对话型：

```text
Header: 对话标题
Messages: Claude 式消息流
Composer: 底部输入框
```

原有运行步骤不要删除业务含义，而是降级为轻状态：

```text
StoryForge
我会先看当前章、前文摘要和人物设定。

正在读取：第三章.md、人物设定.md
```

## 验收标准

完成后，中间 Agent 区应满足：

- 首屏看起来是一个创作会话，而不是任务控制台。
- 顶部只显示对话标题。
- 模型、模式、上下文显示在输入框附近。
- 用户消息右侧小气泡，不显示名称。
- Agent 消息左侧正文块，只显示 `StoryForge`。
- 普通审稿、讨论、方案可以通过自然语言完成。
- 只有写文件、生成 diff、导出时才出现确认。
- 状态信息轻，不压过正文对话。
- 用户不需要寻找“开始审查”按钮；直接说话即可推进。

## 当前结论

StoryForge 桌面端中间 Agent 对话区的产品方向是：

```text
顶部：会话标题
中间：Claude 式自然对话
底部：输入框 + 上下文 + 模型/模式 + 发送
```

核心原则：

所有创作决策都通过对话发生，只有写入文件才需要显式确认。
