# StoryForge 首页输入优先 UI/UX 打磨设计

生成时间：2026-06-01 16:44:20 +08:00

## 1. 背景与目标

StoryForge 首页已经完成 Claude-like 深色创作入口改造，当前结构包含左侧导航、顶部 Provider 状态胶囊、中央标题、创作意图输入框、快捷动作和上下文摘要。用户本次选择的优化范围是 **首页细节打磨**，优先目标是 **使用更顺手**，最终确认方案为 **输入优先**。

本次设计目标是让用户打开首页后第一眼明确主路径：

1. 输入故事想法、章节目标或修订要求。
2. 点击“开始”进入既有 Blueprint 流程。
3. 如用户已经知道目标流程，则使用快捷动作跳转到现有页面。

本次不做移动端专项重构，但必须保留基础响应式能力，确保窄屏不溢出、按钮可换行、输入区可用。

## 2. 范围

### 2.1 本次做

- 强化首页中央输入框的主操作地位。
- 优化“开始”按钮可见性、点击区域和状态层级。
- 将快捷动作从小胶囊调整为更易扫读的辅助入口。
- 降低上下文摘要卡的视觉噪声，避免抢占主输入区注意力。
- 保留真实路由跳转，不新增后端接口。
- 更新或补充 Web 测试，覆盖首页核心结构、真实路由、可访问性和禁止虚假能力文案。

### 2.2 本次不做

- 不实现聊天系统。
- 不实现 AI 意图分类器。
- 不新增 Server Action 或后端创作指令接口。
- 不重构 IDE、Studio、Retrieval、Runs、Artifacts 等页面。
- 不做移动端专项设计；仅保留基础响应式兜底。
- 不引入新的设计系统或图标库。

## 3. 现有实现依据

### 3.1 复用组件

- `apps/web/app/page.tsx`：继续作为薄入口，只渲染 `HomeShell`。
- `apps/web/components/home/HomeShell.tsx`：保留首页整体布局。
- `apps/web/components/home/HomeSidebar.tsx`：保留左侧导航、最近记录和工作区状态。
- `apps/web/components/home/HomeComposer.tsx`：作为本次主优化对象，承载输入框和“开始”按钮。
- `apps/web/components/home/HomeQuickActions.tsx`：保留快捷动作，但调整为辅助路径。
- `apps/web/components/home/HomeContextStrip.tsx`：保留三张状态卡，但降低视觉权重。
- `apps/web/components/home/home-data.ts`：继续作为首页文案、路由和空状态事实源。

### 3.2 技术约定

- 使用 Next.js App Router 与 React 组件组合。
- 仅 `HomeComposer` 这类需要浏览器交互的组件使用 `'use client'`。
- 样式继续采用 Tailwind utility class，不新增全局样式体系。
- 所有可读文案、测试描述和文档使用简体中文。

## 4. 推荐设计

### 4.1 首页视觉层级

首页保持当前深色、低噪声创作入口方向。主层级调整为：

1. 顶部状态胶囊：保留，但视觉权重低，继续链接 `/settings`。
2. 主标题：保留“今天要锻造哪段故事？”。
3. 创作输入框：成为首屏最高优先级交互区域。
4. 快捷动作：位于输入框下方，作为第二路径。
5. 上下文摘要：位于快捷动作下方，作为低权重辅助信息。

### 4.2 输入框

输入框应更像明确的主操作面板：

- 边框从普通暗色边框升级为更清晰的强调边框。
- 聚焦状态增加明显但克制的 ring，帮助用户确认正在输入。
- placeholder 保留现有业务语义，不夸大自动选择流程能力。
- 文本区域高度保持稳定，避免输入和 hover 导致布局跳动。
- 底部工具行保留“附加资料 / 世界观 / 上章摘要”和“创作模式”，但文字尺寸和颜色弱于主按钮。

### 4.3 “开始”按钮

“开始”按钮应成为输入框内唯一强动作：

- 提高最小宽度，保持圆形胶囊形态。
- 使用浅色填充和深色文字，延续现有视觉语言。
- 增加 hover、focus-visible 和 disabled 预留样式。
- 默认行为保持 `router.push('/blueprints')`。
- 不把用户输入提交到后端，避免新增未验证契约。

### 4.4 快捷动作

快捷动作保持真实路由：

- 创建 Blueprint → `/blueprints`
- 启动 BookRun → `/book-runs`
- 审阅并批准 → `/studio`
- 核对证据 → `/retrieval`
- 导出审计 → `/artifacts`

视觉上从较小胶囊调整为更易扫读的辅助按钮：

- 按钮高度略增，点击区域更舒适。
- 图标和文字间距更清楚。
- hover 只轻微强调边框和文字，不压过主输入框。
- 允许换行，桌面端居中排列。

### 4.5 上下文摘要

上下文摘要继续展示：

- 当前作品
- 运行状态
- 下一步建议

调整原则：

- 保持空状态真实，不伪造最近作品或运行成功状态。
- 降低背景对比和阴影感。
- 保持三列布局，但在窄屏自然堆叠。
- 文案继续来自 `homeContextEmpty`，避免散落在组件内。

### 4.6 左侧栏

左侧栏本次仅做轻量一致性打磨：

- 保留现有导航结构和 `homeNavItems`。
- 保留最近记录空状态。
- 保留底部本地工作区块。
- 不新增折叠交互，不引入移动端专项抽屉。

## 5. 交互流程

### 5.1 主流程

1. 用户打开首页。
2. 视觉焦点落在标题和输入框。
3. 用户输入创作意图。
4. 用户点击“开始”。
5. 页面跳转到 `/blueprints`。

### 5.2 辅助流程

1. 用户打开首页。
2. 用户不输入内容，直接选择快捷动作。
3. 快捷动作通过 `Link` 跳转到对应既有路由。

### 5.3 空状态流程

1. 用户尚无最近作品或 BookRun。
2. 最近记录和上下文摘要展示明确空状态。
3. 空状态引导用户先进入 Blueprint。

## 6. 可访问性与响应式

- 主区域继续使用 `main` 和 `aria-labelledby="home-composer-title"`。
- 输入框保留 `label`、`htmlFor` 和 `aria-label`。
- 快捷动作继续使用真实 `Link`，不使用不可访问的点击容器。
- “开始”按钮增加 `focus-visible` 样式。
- 快捷动作允许换行，避免中文按钮在窄屏溢出。
- 不做移动端专项视觉重排，但保持首页在手机宽度下可滚动、可输入、可点击。

## 8. 风险与控制

- **全局样式覆盖风险**：`globals.css` 对 `section`、`nav li` 有默认卡片样式，首页组件已有 `!` 覆盖。本次修改应继续局部控制，不扩大全局覆盖。
- **视觉过重风险**：强调输入框时不能让页面变成高噪声仪表盘；快捷动作和状态卡必须保持辅助层级。
- **契约漂移风险**：不得改变快捷动作路由和默认提交路由。
- **移动端误伤风险**：虽然不做专项设计，但必须保留换行和基本可点击尺寸。

## 9. 验收标准

本次实现完成后应满足：

1. 首页主输入区比当前版本更明确，用户第一眼知道可以输入并开始。
2. “开始”按钮更容易发现和点击。
3. 快捷动作更易扫读，但视觉权重低于输入框。
4. 上下文摘要和最近记录继续使用真实空状态。
5. 不新增后端契约，不伪造能力，不修改非首页页面。
6. Web 测试和 TypeScript 检查通过。
7. `.codex/verification-report.md` 记录本地验证结果、评分和风险。
