# 双模态 AI 小说平台设计

日期：2026-05-12  
状态：已批准进入设计文档阶段  
范围：围绕“生成 -> 精修”闭环的双模态 AI 小说平台产品与系统设计

## 1. 目标

构建一个顶级 AI 小说平台，在同一套底座上同时支持两类核心工作流：

1. 从一个想法自动生成完整、可继续编辑的小说初稿。
2. 对 AI 或人类已有草稿进行深度精修，并保持长篇一致性。

这个平台不能把“生成”和“精修”做成两个割裂产品。两者必须共享同一套世界状态、文风状态、剧情状态、评审历史和版本谱系，让每次改写都能反哺后续生成。

## 2. 产品论点

最强的 AI 小说系统，不是一个超长 Prompt，也不是一个万能 Agent，而是一套“创作操作系统”，它必须具备：

- 以显式创作资产作为真相源
- 可恢复的长任务工作流
- 按场景组装的结构化检索，而不是整本原文硬塞上下文
- 多阶段评审与定向修复，而不是一键重抽整章
- 每次接受生成或改写结果后，自动回写资产状态

本设计选择“资产驱动工作流”作为创作内核，再在其外层包裹平台化能力，以便后续承接协作、分析和商业化。

第一阶段的实现形式应当是**模块化单体**，而不是一开始就拆成分布式微服务。

## 3. 用户

主要用户：

- 想从一个创意快速得到完整初稿的独立作者
- 已经有草稿，需要扩写、重写、统一风格、修复一致性的作者
- 写长篇或系列作品，需要长期世界观和风格控制的重度创作者

次要用户：

- 共同维护同一项目的小型编辑团队
- 关注成本、质量和转化效率的产品运营者

## 4. 成功标准

平台必须稳定做到以下几件事：

1. 根据 premise、大纲和风格设定，生成连贯可读的初稿。
2. 在不破坏角色状态、设定、剧情连续性和文风的前提下精修章节。
3. 将用户批准后的修改自动带入下一章生成。
4. 能解释“为什么这样写”或“为什么这样改”，并展示证据、检索上下文和评审结果。
5. 支持长任务断点续跑、进度追踪和部分结果交付。

## 5. 非目标

第一版明确不做以下事情：

- 打造一个完全自动化的“出版公司工作流”
- 一开始就做学术化、超重型的知识图谱推理系统
- 在工作流和评测闭环没有跑通前，优先投入模型微调
- 从第一天开始就把所有子系统拆成微服务

## 6. 核心产品界面

产品共享一套后端，但面向用户有三个主要界面。

### 6.1 自动成书 Studio

用途：  
把一个想法变成一部完整、可继续修改的小说初稿。

核心步骤：

1. 输入 premise、题材、主题、目标读者、文风目标和约束条件
2. 生成书级大纲、章节计划、场景 beats
3. 基于资产和检索组装 scene packet，逐场景生成正文
4. 在用户审批前执行评审和修复循环
5. 导出为 EPUB、Markdown 等格式

### 6.2 精修工作台 Refinery

用途：  
对已有草稿进行逐章或批量打磨。

核心步骤：

1. 对比源草稿与候选改写结果
2. 对选中片段执行重写、扩写、压缩、润色等操作
3. 执行一致性、文风、可读性检查
4. 批准修改并回写资产变化
5. 让后续生成自动继承这些修改结果

### 6.3 资产中心 Asset Center

用途：  
集中维护“生成”和“精修”都依赖的创作状态。

核心内容：

- 角色
- 地点
- 组织
- 世界观与术语
- 主线与支线
- 伏笔与未回收问题
- 文风规则
- 参考资料
- 评审问题与记忆快照

## 7. 架构总览

系统采用“**B 为核，C 为壳**”：

- 核心：资产驱动工作流引擎，负责规划、写作、检索、评审、修复、回写
- 外壳：平台层能力，负责任务、事件、协作、分析、导出、商业化准备

高层分为三层：

1. 用户体验层
2. 创作智能内核
3. 平台骨架层

## 8. 设计原则

1. 显式状态优于隐藏在上下文里的状态
2. 检索必须为当前场景组装上下文，而不是塞整本书
3. 评审结果必须结构化，而不是只有自然语言建议
4. 修复应只针对有问题的片段，而不是重写健康文本
5. 每次批准后的结果都必须更新未来上下文
6. 平台复杂度必须在创作内核稳定后再增长

## 9. 真相源与资产模型

平台管理的核心不是“文档”，而是“创作资产”。

### 9.1 一级资产

#### Book Graph

表示：

- 书
- 卷
- 章
- 场景
- 段落片段

用途：  
提供结构层级、顺序、进度状态和版本挂载点。

#### Canon Graph

表示：

- 人物
- 地点
- 组织
- 道具
- 术语
- 世界规则

用途：  
维护事实一致性，并在生成和精修时提供相关设定。

#### Plot Graph

表示：

- 主线
- 支线
- 目标
- 冲突
- 伏笔种子
- 回收状态
- 未解 open loops

用途：  
追踪叙事义务和剧情推进状态。

#### Style Profile

表示：

- 叙事视角规则
- 声音标记
- 节奏偏好
- 类型约束
- 禁止模式
- 希望呈现的情绪质感

用途：  
让整本书在长期创作过程中保持统一风格。

#### Draft Lineage

表示：

- 原始草稿
- 重写版本
- 扩写版本
- 压缩版本
- 批准后的最终版本

用途：  
保留修订历史，支持比较、回滚和责任追溯。

#### Evidence Links

表示：

- 某段文本受哪些资产影响
- 使用了哪些检索结果
- 因哪些评审问题触发修复

用途：  
为可解释性、调试和质量分析提供基础。

### 9.2 辅助资产

- 记忆快照
- 章摘要
- 角色状态快照
- 关系变化记录
- 评审问题清单
- Prompt Packs
- 模型运行日志
- 用户意图约束
- 参考资料库

### 9.3 存储模型

关系数据库是业务真相源。

- `PostgreSQL`：存结构化资产、版本、评审问题、任务、权限
- `Vector Index`：只是检索加速器，不是业务真相源
- `Object Storage`：存导出文件、上传资料、快照和辅助产物

## 10. 生成到精修的闭环

这是平台最核心的主流程。

### 10.1 生成链

1. 采集想法和约束
2. 生成 premise、主题和读者定位
3. 生成全书大纲
4. 生成章节计划
5. 生成当前章节的 scene beats
6. 组装 scene packet
7. 生成场景正文
8. 执行评审和修复循环
9. 用户审批或继续调整
10. 回写资产和记忆
11. 下一章生成自动继承已批准状态

### 10.2 精修链

1. 选择目标章节或片段
2. 加载源文本、相关资产和历史状态
3. 应用精修模式：重写、扩写、压缩、文风润色、设定修复、一致性修复
4. 评审结果
5. 必要时生成结构化问题单
6. 仅对失败片段执行定向修复
7. 重新评审
8. 用户批准
9. 回写设定、剧情、风格漂移和记忆变化

### 10.3 核心规则

“生成”和“精修”不是两个孤立系统，而是同一套资产闭环的两个入口。

## 11. Agent 拓扑

系统使用一组边界清晰的专用 Agent。

### 11.1 Book Director

职责：

- 把想法变成 premise 和全书战略
- 制定主题、题材和长程结构目标

### 11.2 Scene Architect

职责：

- 把章节计划拆成场景 beats
- 控制冲突节奏和情绪推进

### 11.3 Draft Writer

职责：

- 基于 scene packet 写出可读正文
- 遵守风格和一致性约束

### 11.4 Canon Librarian

职责：

- 解析设定引用
- 维护角色、地点、术语等状态

### 11.5 Style Conductor

职责：

- 约束文风、节奏、视角与反模式

### 11.6 Continuity Judge

职责：

- 发现设定冲突、状态漂移、伏笔遗漏和推进断裂

### 11.7 Prose Critic

职责：

- 发现重复、平、AI 味、出戏表达和可读性问题

### 11.8 Repair Agent

职责：

- 根据结构化问题单，仅修复失败片段

### 11.9 Agent 边界规则

默认情况下，不应把规划、写作、评审、修复全部合并成一个万能 Agent。分工越清楚，越利于控制、缓存、分析与恢复。

## 12. 检索策略

检索的目标不是“搜几段相关原文”，而是产出一个高质量的 **scene packet**。

### 12.1 固定槽位

- 当前章节目标
- 活跃角色
- 当前关系状态
- 未回收 open loops
- 文风和 POV 规则
- 必须包含的事实
- 必须规避的事实
- 用户本次任务意图

### 12.2 动态槽位

- 相似历史场景
- 相关设定条目
- 相关伏笔提及
- 邻近章节摘要
- 参考资料
- 已写草稿片段

### 12.3 推荐检索链

1. 根据当前任务和场景目标进行查询规划
2. 用图结构和业务条件做候选过滤
3. 执行 dense search 与 sparse search
4. 进行 rerank
5. 在 token 预算内完成上下文组装

### 12.4 检索要求

- 日常操作不能默认检索整本书
- 优先使用摘要和状态快照，其次才是原始长文本
- 把宝贵 token 留给创作推理，而不是背景搬运

## 13. 评审与修复闭环

平台必须把“写得好”变成可执行的工程过程。

### 13.1 评审层次

#### 第 1 层：规则检查

例如：

- 命名不一致
- POV 违规
- 术语违规
- 长度和结构约束
- 禁用表达

#### 第 2 层：LLM Judges

例如：

- 角色一致性
- 情绪连贯性
- 节奏质量
- 文风贴合度
- 可读性
- 伏笔完整性

#### 第 3 层：Diff Repair

只改失败片段，不动健康文本。

### 13.2 结构化问题单

每个问题至少包含：

- 类别
- 严重级别
- 片段位置
- 简短说明
- 证据引用
- 推荐修复模式

### 13.3 重新评审规则

每次自动修复后，必须重新评审，合格后才允许进入用户审批。

## 14. 平台骨架层

这些能力包裹创作内核，但不拥有创作真相源。

### 14.1 任务中心 Job Center

负责：

- 长任务运行
- 批处理
- 断点续跑
- 重试策略
- 进度追踪

### 14.2 事件流 Event Stream

负责：

- 任务状态变化
- 评审完成通知
- 审计日志
- 协作通知

### 14.3 协作层 Collaboration

负责：

- 评论
- 版本审批
- 角色权限
- 审核流

### 14.4 分析层 Analytics

跟踪：

- 成本
- 延迟
- 接受率
- 修复率
- Judge 失败类别
- 项目完成行为

### 14.5 Provider Gateway

统一接入：

- LLM
- Embedding
- Reranker
- 图片生成或封面生成能力

## 15. 技术选型

推荐基线：

- 前端：`Next.js + React`
- 富文本与对比编辑：`TipTap + CodeMirror`
- 应用 API：`FastAPI`
- 工作流 Runtime：`Python + LangGraph`
- 主数据库：`PostgreSQL`
- 初始向量检索：`pgvector`
- 队列与缓存：`Redis`
- 对象存储：`S3 兼容存储`
- 可观测性：`Langfuse + OpenTelemetry`

原因：

- React 适合复杂编辑器和多面板交互
- FastAPI 和 Python 很适合 LLM 编排、工作流和数据 API
- LangGraph 适合长任务 checkpoint 和断点恢复
- PostgreSQL + pgvector 在第一阶段足够强，也能控制运维复杂度

## 16. 系统边界

### 16.1 同步交互

适合：

- 短改写
- 段落扩写
- 实时检索预览
- 左右对比查看
- 快速资产编辑

### 16.2 异步任务

适合：

- 整章生成
- 整本生成
- 批量精修
- 多轮评审
- 大体量导出
- 批量 embedding 刷新

### 16.3 服务边界

- Web API 不直接编排复杂多步写作流程
- Workflow Runtime 只读写任务、资产和事件，不直接操纵前端会话状态
- 检索索引只负责加速，不取代结构化真相源

## 17. 评测与质量系统

平台必须从第一阶段就带上评测闭环。

### 17.1 自动评测

跟踪：

- 一致性错误率
- 修复成功率
- 用户接受率
- 源文本与批准文本的 edit distance
- 重复表达频率
- 未回收 open loop 数量

### 17.2 人类反馈信号

跟踪：

- 哪些建议被接受
- 哪些 AI 结果被用户手工重写
- 哪些 Judge 类别最容易导致拒绝

### 17.3 评测资产

维护一组可复用基准集：

- 短场景基准
- 长章节基准
- Canon 压测集
- 风格一致性测试集

## 18. MVP

MVP 不是证明“会生成”，而是证明整个闭环成立。

MVP 范围：

1. 单用户项目
2. 书、章、场景、角色、设定、风格、open loop 等基础资产
3. 从大纲到章节再到场景的生成链
4. 包含重写、扩写、润色的精修工作台
5. 结构化 Judge 和定向 Repair
6. 用户批准后自动回写资产
7. EPUB 和 Markdown 导出

MVP 完成定义：

- 用户能够生成一章
- 对这一章进行精修
- 批准修改
- 下一章生成自动继承这些修改结果

MVP 明确不包含：

- 多人协作
- 计费和商业化能力
- 插件市场或第三方扩展框架
- 全量事件驱动的服务拆分

## 19. 演进路线

### Phase 1

- 跑通强闭环
- 稳定资产模型
- 任务可恢复
- 基础评测体系

### Phase 2

- 系列级记忆
- 完整世界观中心
- 批量精修
- 风格包复用
- 更丰富的质量看板

### Phase 3

- 团队工作区
- 协作审批
- 商业化控制
- 事件驱动分析扩展
- 可插拔模型接入层

## 20. 风险与缓解

### 风险 1：过早平台化

缓解：  
优先把创作内核、任务可靠性和评测闭环打稳，再扩 SaaS 能力。

### 风险 2：检索噪音污染正文质量

缓解：  
坚持 scene packet 组装、摘要优先、过滤和 rerank，避免整本原文堆上下文。

### 风险 3：Judge 意见太虚，无法自动修复

缓解：  
强制输出结构化问题单，并绑定 span 级位置和修复模式。

### 风险 4：资产维护成本过高

缓解：  
把资产回写尽量自动化，初始图结构坚持“够用主义”。

## 22. 外部优秀方案吸收决策

本章把外部产品、开源工程、研究论文和官方技术文档转化为 StoryForge 的实施决策。所有外部方案只吸收可验证的设计模式，不复制外部代码；进入代码阶段前必须再次核验许可证、维护状态、接口边界和本地验证方式。

### 22.1 产品方案吸收

- [Sudowrite Story Bible](https://docs.sudowrite.com/using-sudowrite/1ow1qkGqof9rtcyGnrWUBS/what-is-story-bible/jmWepHcQdJetNrE991fjJC)：吸收“从想法到大纲、角色、世界观、场景和正文”的向导式创作链。StoryForge 中对应为 Book Director 到 Scene Architect 的前置规划流程。
- [Sudowrite Chapter Continuity](https://docs.sudowrite.com/using-sudowrite/1ow1qkGqof9rtcyGnrWUBS/chapter-continuity/4KL8gFeLZQ6GSBjDWtSbV6)：吸收章节连续性显式管理机制。StoryForge 必须记录上一章摘要、角色状态变化、未回收伏笔和下一章继承约束。
- [Novelcrafter Codex](https://docs.novelcrafter.com/en/collections/7255877-the-codex)：吸收 Codex 作为作者可编辑资产中心的产品形态。StoryForge 的 Asset Center 必须让角色、地点、组织、物件、术语、子情节和系列共享设定成为一等资产。

### 22.2 开源工程方案吸收

- [InkOS](https://github.com/Narcooo/inkos)：吸收“长期真相文件、多 Agent 分工、审计失败后修订、人工审核门、章节快照和回滚”的工程闭环。StoryForge 不照搬 Markdown 真相文件实现，而是映射为结构化资产表、任务事件、评审问题单和版本谱系。
- [autonovel](https://github.com/NousResearch/autonovel)：吸收“基础设定 -> 初稿 -> 自动修订 -> 导出”的阶段门禁，以及章节评分不达标时重试或修订的控制方式。StoryForge 第一阶段只吸收生成与修订门禁，不吸收有声书、封面和营销落地页。
- [NovelGenerator](https://github.com/KazKozDev/NovelGenerator)：吸收前端体验线索，包括 Agent 活动日志、进度条、流式章节显示、版本差异查看、自动保存状态和多格式导出提示。

### 22.3 研究方法吸收

- [Re3](https://arxiv.org/abs/2210.06774)：吸收递归重提示、分段生成、多候选排序和一致性修订思想。StoryForge 中对应为章节/场景级生成候选、Judge 评分和 Repair 循环。
- [DOC](https://arxiv.org/abs/2212.10077)：吸收详细大纲控制，要求正文生成严格绑定章节目标、场景 beats 和关键事实，而不是只依赖自由生成。
- [DOME](https://arxiv.org/abs/2412.13575)：吸收动态层级大纲和记忆增强思想，让大纲、记忆和资产状态随批准文本持续更新。
- [StoryWriter](https://arxiv.org/abs/2506.16445)：吸收 Outline Agent、Planning Agent、Writing Agent 的分层协作方式，但第一阶段保持可观测、可恢复的少量专用 Agent，不做完全自治的复杂 Agent 社会。

### 22.4 官方技术底座吸收

- [LangGraph durable execution](https://docs.langchain.com/oss/python/langgraph/durable-execution)：作为长任务 checkpoint、失败恢复和可恢复工作流的候选底座。
- [LangGraph interrupts](https://docs.langchain.com/oss/python/langgraph/interrupts)：作为用户审批、人工审核门和高影响动作确认的候选机制。
- [FastAPI 大型应用组织](https://fastapi.tiangolo.com/tutorial/bigger-applications/)：作为模块化单体 API 的组织参考，后续按资产、生成、精修、评审、任务和导出拆分 `APIRouter`。
- [Next.js App Router](https://nextjs.org/docs/app)：作为复杂编辑器、资产中心、任务中心和多面板工作台的前端组织参考。

### 22.5 Phase 1 吸收边界

第一阶段必须吸收以下机制：

1. 可编辑的资产真相源：角色、设定、剧情、风格、伏笔、章节状态和版本谱系。
2. 章节连续性记录：上一章摘要、状态变化、未回收问题、下一章继承约束。
3. 层级大纲控制：全书大纲、章节计划、场景 beats 和关键事实逐级落地。
4. Scene Packet：每次生成或精修只装配当前片段需要的资产、摘要、检索证据和风格规则。
5. 结构化 Judge：问题必须包含类别、严重级别、位置、证据、建议修复模式和重新评审结果。
6. 定向 Repair：只修复失败片段，不整章重抽健康文本。
7. 可恢复任务：长任务必须具备状态持久化、断点续跑、审批中断和失败重试记录。

### 22.6 暂不吸收项

以下能力不进入第一阶段，除非后续验证证明它们是闭环成立的必要条件：

- 完整 SaaS 协作、计费、团队工作区和商业化控制。
- 插件市场、第三方扩展框架和复杂权限治理。
- 有声书、封面生成、营销落地页和出版公司工作流。
- 全量微服务、独立事件总线平台和过重基础设施拆分。
- 学术化重型知识图谱、全自动关系推理和不可解释的黑箱 Agent 社会。

### 22.7 验收链路更新

吸收外部方案后，MVP 验收必须新增一条可重复证明链：

1. 从用户输入生成全书大纲、章节计划和当前场景 beats。
2. 组装 Scene Packet，并记录其中使用的资产、摘要、检索结果和上下文预算。
3. 生成章节片段，执行结构化 Judge，产出问题单。
4. 对失败片段执行 Repair，并重新评审。
5. 用户批准结果后，生成资产差异、章节连续性记录和版本谱系记录。
6. 下一章生成读取上述批准状态，并在验证报告中证明继承关系。

## 23. AGENTS 执行治理与交付门禁

本计划的后续所有设计、实现、验证和交付必须遵循仓库根目录 `AGENTS.md`。本章是对产品计划的执行约束，不替代前文的产品架构和技术设计。

### 23.1 语言与文档规则

- 所有新增计划、设计、日志、验证报告、测试描述、错误提示和提交说明必须使用简体中文。
- 技术名、库名、代码标识符和协议字段可保留英文原名，但解释文本必须使用简体中文。
- 中文主规格是唯一事实源；重复文档只能作为中文引用入口，不能继续维护英文镜像正文。
- 所有任务过程文件必须写入项目本地 `.codex/`，包括上下文摘要、操作日志和验证报告。

### 23.2 强制执行顺序

每次进入实现或文档改动前，必须按以下顺序执行：

1. 使用 `sequential-thinking` 明确任务目标、风险、依赖和验收条件。
2. 使用 `shrimp-task-manager` 拆分或确认任务，形成可验证的执行单元。
3. 使用本地工具直接执行修改，优先使用 `desktop-commander` 完成本地文件分析、搜索和数据处理。
4. 对涉及库、框架、SDK 或 API 的技术判断，优先使用 Context7 查询官方文档。
5. 需要开源实现参考时，优先使用 `github.search_code`；若工具不可用，必须在 `.codex/operations-log.md` 记录限制与替代方案。

### 23.3 编码前上下文门禁

任何代码实现前必须生成 `.codex/context-summary-[任务名].md`，至少包含：

- 3 个相似实现或模式的路径、用途和可复用点。
- 当前任务的依赖与集成点，包括输入、输出、配置和环境需求。
- 现有测试框架、命名约定、格式化规则和本次测试策略。
- 可复用组件清单，以及不重复造轮子的证据。
- 主要风险点，包括并发、边界、性能、数据一致性和验证缺口。

如果仓库仍处于文档阶段、没有代码或测试文件，摘要必须明确说明缺口，并把补齐代码结构、测试框架和验证脚本列为进入实现阶段的前置条件。

### 23.4 阶段化交付计划

后续实施必须按阶段门禁推进：

1. **研究阶段**：阅读 `AGENTS.md`、本计划、相关草案和官方文档；禁止直接编码。
2. **计划阶段**：形成任务拆分、接口契约、验收标准、验证命令和回滚说明。
3. **实施阶段**：按模块化单体边界小步修改；优先复用成熟 SDK、官方库和既有模块。
4. **验证阶段**：运行本地测试、文档检查或可重复脚本，并记录命令、输出摘要和失败处理。
5. **交付阶段**：更新 `.codex/verification-report.md`，给出评分、建议、风险和后续补偿计划。

### 23.5 交付物映射

第一阶段交付物必须能映射到明确文件或验证证据：

- **产品与架构文档**：维护在 `docs/superpowers/specs/`，以中文主规格为准。
- **上下文摘要**：维护在 `.codex/context-summary-[任务名].md`。
- **操作日志**：维护在 `.codex/operations-log.md`，记录工具限制、关键决策和偏离原因。
- **验证报告**：维护在 `.codex/verification-report.md`，记录本地验证结果和评分。
- **实现代码与测试**：进入代码阶段后必须补齐自动化测试；没有测试不得声明实现完成。

### 23.6 本地验证要求

所有验证必须由本地 AI 自动执行，不能依赖 CI、远程流水线或人工外包验证。验证至少覆盖：

- 文档改动：检查目标章节、关键约束、语言规范和链接引用。
- 代码改动：运行项目既定单元测试、冒烟测试、格式化或静态检查。
- 工作流改动：验证正常流程、边界条件、失败恢复和断点续跑。
- AI 编排改动：验证结构化输入输出、问题单格式、重试策略和资产回写结果。

若工具缺失或覆盖不足，必须在验证报告中记录原因、风险等级、补偿计划和截止条件。

### 23.7 质量审查评分

每次交付前必须使用 `sequential-thinking` 做审查，并在 `.codex/verification-report.md` 输出：

- 技术维度评分：代码质量、测试覆盖、规范遵循。
- 战略维度评分：需求匹配、架构一致、风险评估。
- 综合评分：0-100 分。
- 明确建议：通过、退回或需讨论。

综合评分达到 90 分且建议为“通过”时，才允许确认交付。低于 80 分且建议为“退回”时，必须回到计划或实施阶段修正。

### 23.8 MVP 验收重定义

本计划中的 MVP 不是占位实现，也不是最小演示。第一阶段验收必须证明完整闭环：

1. 用户可以创建单用户项目和基础创作资产。
2. 系统可以从大纲生成一章，并保留可解释的 Scene Packet 和证据链接。
3. 用户可以对该章执行精修、结构化评审和定向修复。
4. 用户批准后，系统可以回写资产、剧情、风格和记忆变化。
5. 下一章生成必须能读取并体现已批准修改。
6. 本地验证报告必须记录以上链路的可重复验证步骤。

### 23.9 破坏性变更与复用策略

实现阶段优先采用标准化和生态复用方案。除非已有库或既有模块无法满足需求，并在操作日志中记录特例批准，否则不得新增自研替代方案。

若后续发现早期设计与实际实现冲突，应采用清晰的破坏性迁移策略，不为未落地的旧接口做兼容层；同时必须提供迁移步骤、回滚方案和验证脚本。

## 24. 最终建议

下一步应当按“**同一套资产底座上的双入口系统**”推进：

- 入口一：自动成书
- 入口二：精修工作台

第一优先级不是“接最强模型”、不是“做最炫图谱”、也不是“先补全 SaaS 外壳”，而是先把以下五件事做强：

1. 共享资产真相源
2. Scene Packet 检索
3. 结构化评审
4. 定向修复
5. 可靠回写到未来上下文

只要这五件事做强，这个平台就会有真正的质量壁垒，并且有机会成为同类 AI 小说产品里最强的一档。
