# StoryForge AI 持续迭代指导文件

本文件用于指导 AI 持续推进 StoryForge 项目。

项目路径：

```text
D:\StoryForge\1-renovel-ai-ai-rag-tavern
```

每次让 AI 参与开发、修 bug、整理文档、优化结构、测试验证、Git 维护或规划任务前，都应先阅读本文件。

---

## 0. 与 AGENTS.md 的关系

项目级总规范位于：

```text
D:\StoryForge\AGENTS.md
```

本文件是 `docs/internal/AGENTS.md` 在当前项目中的日常执行版。

执行规则：

- `docs/internal/AGENTS.md` 是上位规范，本文件负责把它落成可重复执行的流程。
- 如两者冲突，优先遵循 `docs/internal/AGENTS.md`。
- 所有回复、文档、注释、日志、提交信息默认使用简体中文。
- 代码标识符、命令、包名、API 名称按项目和生态习惯保留英文。
- 所有结论必须有证据，禁止凭感觉改代码。
- 所有验证必须在本地执行，或给出可复现的本地验证步骤。

## 1. 项目推进目标

AI 的目标不是一次性“做完整个项目”，而是让项目稳定、连续、可追踪地往前走。

核心原则：

```text
找问题
查上下文
找优质轮子
小步推进
本地验证
总结留痕
维护 Git
继续下一轮
```

每次工作都必须做到：

- 小步快跑，不做无关大改。
- 先稳后快，先闭环再扩展。
- 先修主线，再加功能。
- 复用优先，避免重复造轮子。
- 测试优先，不能验证就不能声称完成。
- 结果必须落到文件、测试、日志、TODO 或 Git 状态中。

## 3. 固定迭代循环

本项目的长期推进流程固定为：

```text
读规则 → 找问题 → 查上下文 → 搜索优质轮子 → 计划 → 执行 → 测试验证 → 质量审查 → 总结留痕 → Git 维护 → 继续推进
```

简化记忆版：

```text
找问题 → 找轮子 → 继续 → 测试 → 总结 → Git → 继续
```

每次 AI 接手项目时，都应从这个循环中继续，而不是重新开始。

## 5. 找问题阶段

AI 在继续推进前，必须先找问题，而不是直接写代码。

优先检查：

```text
TODO.md 中是否有阻塞项
README.md 与实际代码是否不一致
package.json 中脚本是否可用
最近修改是否遗留未完成内容
测试是否缺失或失败
文档是否缺启动、配置、验证说明
核心功能链路是否断裂
目录结构是否明显混乱
Git 工作区是否已有未提交改动
是否存在重复实现或自研复杂逻辑
```

必须使用这个自检提示词：

```text
你是否百分之百有把握已完成 `XXX`？

如果没有，请找出所有可能的漏洞、遗漏、风险和未验证点，提出合理的解决方案，并重复这个过程，直到你确信无疑为止。
```

`XXX` 替换为当前任务、当前功能、当前修复、当前文档目标或本轮目标。

找问题阶段要循环自查：

```text
提出完成判断 → 找漏洞 → 给解决方案 → 修正或记录 → 再次判断 → 直到无明显漏洞
```

输出格式：

```md
## 问题扫描

自检目标：

- `XXX`

是否百分之百有把握：

- 是 / 否

## 发现的漏洞、遗漏、风险和未验证点：

## 解决方案：

## 本轮建议处理：

## 暂不处理但必须记录的问题：

## 再次自检结论：
```

如果发现问题太多，不能全部处理，必须按影响程度排序，并只选择 1 到 3 个进入本轮推进。

## 6. 查上下文阶段

编码前必须先查上下文。

简单文档任务可以轻量执行；涉及代码、接口、配置、构建、测试、架构的任务必须完整执行。

动手前至少确认：

- 当前任务影响哪些文件。
- 是否已有相似实现。
- 是否已有测试模式。
- 是否已有可复用工具、模块或依赖。
- 当前项目命名、目录和导入风格。
- 本轮如何验证。
  涉及代码修改时，必须尽量完成：

```text
1. 搜索相关文件
2. 搜索关键实现
3. 阅读至少 3 个相似实现或相近模式
4. 检查现有测试文件
5. 检查可复用工具、helpers、utils、packages
6. 查询相关库或框架官方文档
7. 提取项目模式与风险点
```

如果找不到 3 个相似实现，应说明实际找到多少、为什么不足、用什么替代依据。

中等及以上任务，或任何跨模块代码任务，必须生成或更新：

```text
.codex/context-summary-[任务名].md
```

摘要至少包含：

- 任务目标
- 相似实现分析
- 可复用组件清单
- 项目约定
- 测试策略
- 依赖与集成点
- 风险点
- 验证方式

## 7. 搜索优质轮子阶段

在实现功能、修复复杂问题或优化结构前，AI 必须判断是否需要搜索已有优质方案。

流程位置：

```text
找问题 → 查上下文 → 搜索优质轮子 → 计划 → 执行
```

以下情况必须优先搜索或评估已有方案：

- 需要新增通用能力，例如日志、配置、校验、状态管理、解析器、缓存、队列、上传、测试工具。
- 当前代码里存在明显自研复杂逻辑。
- 要实现的功能在生态中已有成熟方案。
- 涉及框架、库、SDK、API 的最佳实践。
- 不确定当前实现是否符合主流做法。
- 需要减少维护成本和长期技术债务。
- 需要替换重复实现或整理公共能力。

搜索优先级固定为：

```text
1. 项目内部已有实现
2. 当前项目依赖中已有能力
3. 官方文档和 SDK
4. 成熟开源库
5. 社区最佳实践
6. 自研实现
```

自研只能作为最后选择。若必须自研，应记录原因和后续替换条件。

工具选择：

- 本地文件搜索与分析优先使用 desktop-commander。
- 编程库、框架、SDK、API 文档优先使用 context7。
- 开源实现参考优先使用 GitHub 代码搜索。
- 网络资料只能作为补充，不能替代本地代码和官方文档。

轮子评估输出：

```md
## 轮子评估

是否需要引入或复用现有方案：

- 是 / 否

已检查：

- 项目内部已有实现：
- 当前依赖能力：
- 官方或成熟方案：
- 开源参考：

## 推荐方案：

## 不推荐方案：

## 理由：

是否需要新增依赖：

- 是 / 否

如果新增依赖，影响评估：

- 包体积：
- 维护成本：
- 兼容性：
- 替换现有代码成本：
- 本地验证方式：
```

轮子使用原则：

```text
能复用项目已有能力，就不新增依赖。
能用成熟库，就不写复杂自研。
简单问题不要过度引入大库。
新增依赖必须说明收益大于成本。
复用方案必须写清楚使用方式和验证方式。
```

## 8. 任务拆分与选择

不要让 AI 一次性执行以下任务：

- 完善整个项目
- 重构全部代码
- 修复所有 bug
- 做完整系统
- 优化所有文档
- 整理所有结构

每次只能选择 1 到 3 个小任务。

合适的小任务示例：

- 新建一个配置文件
- 修复一个具体接口
- 增加一个具体页面
- 优化一个函数
- 补充一段文档
- 增加一个测试
- 整理一个目录说明
- 删除一个确认废弃的实现
- 替换一个重复自研工具

每个任务必须满足：

```text
目标明确
影响范围小
可以验证
可以记录结果
```

任务优先级：

```text
阻塞主线 > 明确 bug > 测试缺口 > 文档缺口 > 重复造轮子 > 结构优化 > 新功能
```

## 10. 执行阶段

执行时必须遵守：

- 只修改和本轮任务直接相关的文件。
- 禁止顺手大范围改结构。
- 禁止同时引入多个无关功能。
- 禁止修改无法解释的代码。
- 优先复用已有模块、工具和依赖。
- 遵循项目既有命名、目录、导入、测试和格式风格。
- 所有中文说明、注释、文档、日志必须使用简体中文。

## 11. 测试验证阶段

每次修改后必须尽可能进行本地验证。

优先使用项目已有命令：

```powershell
pnpm test
pnpm lint
pnpm build
```

如存在更小范围命令，应优先运行更相关的验证：

```powershell
pnpm --filter <app-or-package> test
pnpm --filter <app-or-package> build
pnpm --filter <app-or-package> lint
```

验证失败时：

- 不得声称完成。
- 必须记录失败命令、失败摘要、可能原因。
- 判断是继续修复，还是写入 `docs/internal/TODO.md` 作为后续任务。
- 连续三次验证失败，必须暂停实现，回到找问题和计划阶段。

如果命令不存在或工具缺失：

- 说明缺失原因。
- 给出替代验证方式。
- 在 `docs/internal/TODO.md` 或 `.codex/verification-report.md` 记录补足计划。

## 14. Git 维护阶段

每轮修改后必须检查 Git 状态。

使用：

```powershell
git status --short
git diff --stat
git diff
```

AI 必须说明：

- 当前有哪些未提交修改。
- 每个修改属于哪个任务。
- 是否存在不应提交的临时文件。
- 是否建议提交。
- 建议提交信息是什么。

如果用户没有明确要求提交，AI 不应自动提交。

如果用户要求提交，提交前必须确认：

1. 验证已经完成，或失败原因已记录。
2. `docs/internal/TODO.md` 已更新。
3. `.codex/operations-log.md` 或验证报告已按需更新。
4. 没有无关文件混入。
5. 提交信息为简体中文。

提交信息格式：

```text
类型：简短说明
```

示例：

```text
文档：优化 AI 持续迭代指导流程
修复：修复知识库上传接口异常
测试：补充角色卡解析测试
重构：复用成熟方案整理配置读取逻辑
```

常用类型：

```text
功能
修复
文档
测试
重构
构建
整理
```

## 15. Git 分支与提交纪律

如果当前改动较小，可以在当前分支继续。

如果是较大的功能、重构或风险较高的修复，应建议新建分支：

```powershell
git checkout -b work/<short-task-name>
```

分支命名示例：

```text
work/todo-guide
work/fix-upload-api
work/add-rag-tests
work/docs-cleanup
work/replace-custom-parser
```

一次提交只对应一个明确主题。

不要把无关修改混在一个提交里。

以下内容默认不应提交：

- 临时调试文件
- 本地缓存
- 私有环境变量
- 大型生成产物
- 与本轮任务无关的格式化改动
- 未解释来源的文件

如果确实需要提交生成产物，必须说明原因。

## 18. 禁止事项

AI 在推进项目时禁止：

- 不读规则和项目状态直接修改。
- 不找问题直接编码。
- 不查上下文直接编码。
- 不评估优质轮子就写复杂自研逻辑。
- 一次性大范围重构。
- 同时处理大量无关任务。
- 修改后不测试。
- 测试失败却声称完成。
- 不更新 `docs/internal/TODO.md`。
- 不检查 Git 状态。
- 自动提交未经用户确认的改动。
- 把临时文件、缓存、私密配置提交进仓库。
- 只聊天不落地结果。
- 凭猜测下结论，不提供证据。
